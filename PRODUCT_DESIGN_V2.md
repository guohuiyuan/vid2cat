# vid2cat V2 Design

## 1. 新方向概述

参考 `项目设计书2.md`，`vid2cat` 建议从“单条抖音 -> 单个猫咪图鉴”的内容分析产品，升级为“用户领取一只动漫猫 -> 连续喂 1~5 条抖音链接 -> 逐步改变属性 -> 最终生成专属猫形象 -> 分享到猫咪广场”的养成型 Web 应用。

这意味着项目的核心对象不再是 `atlas`，而是 `cat`。

新的产品主循环：

1. 用户注册并领取一只初始动漫猫。
2. 用户给猫喂抖音链接，每只猫最多喂 5 次。
3. 每次喂链接后，系统解析视频内容，生成摘要和属性增减。
4. 属性累积后，猫的名字、编号、背景故事、外形设定逐步变化。
5. 满足条件后生成最终猫形象。
6. 用户可与自己的猫对话；对话中如果包含抖音链接，也会触发一次“喂养事件”。
7. 用户可将猫分享到猫咪广场，其他用户可以浏览、点赞、评论。

## 2. 产品目标

### 2.1 用户视角

- 每个用户拥有一只自己的猫。
- 每条抖音链接都会真实影响猫的成长方向。
- 猫有明确的养成终点，而不是无限堆积内容。
- 养成结果可展示、可分享、可被别人互动。

### 2.2 管理员视角

- 可配置不同阶段的提示词预设。
- 可配置喂养规则、属性映射规则、生成图规则。
- 可查看用户猫的成长记录与异常任务。

### 2.3 技术视角

- 继续使用 `FastAPI + Jinja2 + SQLite + Node.js/PicGo` 即可。
- 保持单体应用结构，优先保证功能闭环。
- 所有运行期数据统一落在 `data/` 下，方便 Docker 单卷挂载与备份。

## 3. 核心玩法设计

### 3.1 领取初始猫

每位用户注册成功后，自动分配一只“初始态动漫猫”。

初始猫建议包含：

- 猫编号
- 临时名字
- 当前阶段
- 当前喂养次数
- 当前形象状态
- 六维属性初始值
- 初始背景故事

### 3.2 喂抖音链接

每次喂养输入：

- 一条抖音作品链接
- 可选备注

系统处理：

1. 解析抖音内容。
2. 模型 1 输出视频摘要、关键词、风格标签、情绪/主题判断。
3. 规则层将视频内容映射成属性增减。
4. 记录本次喂养结果。
5. 刷新猫当前状态。

约束：

- 每只猫最多喂 5 次。
- 同一作品建议去重，避免刷属性。
- 喂养成功后要展示“本次带来的属性变化”和“累计成长变化”。

### 3.3 对话玩法

用户可与自己的猫聊天。

规则：

- 普通文本：只进入对话，不改变属性。
- 对话中含抖音链接：走“聊天触发喂养”的特殊分支。
- 如果已喂满 5 次，则仅提示不能再加属性，但仍可普通聊天。

### 3.4 最终生成

达到以下任一条件后可生成最终形象：

- 已喂满 5 条链接。
- 管理员允许提前生成。

最终生成过程：

1. 汇总 1~5 条视频的摘要、标签、属性变化记录。
2. 模型 2 生成完整的猫设定：
   - 名字
   - 编号
   - 品种/外观
   - 六维属性说明
   - 综合能量
   - 背景故事
   - 性格
   - 口头禅
3. 模型 3 生成最终动漫猫形象。
4. 上传图床，数据库只保存最终 URL。

### 3.5 猫咪广场

用户可将自己的猫发布到广场。

广场页展示：

- 猫图
- 编号
- 名字
- 当前阶段或最终态
- 六维属性摘要
- 综合能量
- 最近一次成长时间
- 点赞数
- 评论数

支持互动：

- 浏览
- 点赞
- 评论

## 4. 属性系统设计

根据设计书，建议属性固定为 6 个主维度 + 1 个综合值：

- 智慧
- 毅力
- 创造
- 灵敏
- 协作
- 能量
- 综合能量

说明：

- `综合能量` 不建议直接由用户输入，而是由六维属性按规则计算。
- 每次视频喂养产生一组属性变化量，例如 `+3 智慧, -1 协作, +2 创造`。
- 后续页面上可以同时展示：
  - 当前属性值
  - 最近一次变化
  - 五次累计成长轨迹

## 5. 页面设计建议

### 5.1 首页

建议从“解析页”改为“领取猫 / 进入我的猫”入口页。

模块：

- 产品介绍
- 我的猫入口
- 猫咪广场入口
- 登录/注册入口

### 5.2 我的猫页

这是新的主页面。

布局建议：

- 左侧：猫形象、名字、编号、阶段、喂养次数
- 右侧：六维属性、综合能量、背景故事、最近成长记录

操作区：

- 输入抖音链接喂猫
- 与猫对话
- 查看历史喂养记录
- 生成最终形象
- 发布到广场

### 5.3 喂养记录页

展示每次喂养的：

- 视频标题
- 视频摘要
- 属性变化
- 喂养时间
- 链接来源

### 5.4 广场页

列表卡片展示：

- 猫图
- 编号
- 名字
- 综合能量
- 六维属性缩略
- 点赞/评论统计

### 5.5 广场详情页

展示：

- 猫完整形象
- 最终背景故事
- 五次喂养轨迹摘要
- 评论区

## 6. 模型职责重划分

### 6.1 模型 1：视频解析与属性变化

输入：

- 抖音标题
- 描述
- 作者信息
- 标签
- 可能的字幕/正文

输出建议：

- 视频内容摘要
- 主题标签
- 情绪标签
- 对 6 维属性的增减建议
- 对本次成长阶段的解释

这是 V2 中最关键的模型。

### 6.2 模型 2：猫设定与对话人格

输入：

- 当前猫已有属性
- 历史喂养摘要
- 当前成长阶段
- 管理员预设提示词

输出：

- 猫名字
- 编号风格
- 外貌描述
- 背景故事
- 对话风格
- 最终绘图提示词骨架

### 6.3 模型 3：最终猫图生成

输入：

- 模型 2 输出的人设和绘图提示词
- 可能的风格预设

输出：

- 最终猫图
- 变体图
- 最终图传 URL

## 7. 数据目录约束

V2 需要把所有运行期数据统一放到 `data/` 下。

建议目录结构：

```text
vid2cat/
├─ data/
│  ├─ app.db
│  ├─ config/
│  │  ├─ app.env
│  │  ├─ picgo.config.json
│  │  └─ presets.json
│  ├─ uploads/
│  │  ├─ raw/
│  │  └─ mirrored/
│  ├─ generated/
│  │  ├─ cat_images/
│  │  └─ temp/
│  ├─ exports/
│  ├─ cache/
│  │  ├─ douyin/
│  │  └─ model_responses/
│  ├─ logs/
│  └─ admin_uploads/
├─ src/
├─ scripts/
└─ ...
```

### 7.1 必须进入 `data/` 的内容

- SQLite 数据库
- 上传中转文件
- 图像生成临时文件
- 缓存
- 导出文件
- 日志
- 管理员上传测试图片
- PicGo 运行配置
- 提示词预设配置

### 7.2 不应进入 `data/` 的内容

- Python 源码
- HTML 模板
- CSS/JS 静态资源
- `pyproject.toml`
- `package.json`

## 8. Docker 挂载策略

目标是做到只挂一个卷就能保住所有业务数据。

建议容器内固定路径：

- 应用目录：`/app`
- 数据目录：`/app/data`

推荐挂载：

```yaml
services:
  vid2cat:
    volumes:
      - ./data:/app/data
```

如果需要环境变量文件，也建议迁入：

- 宿主机：`./data/config/app.env`
- 容器内：`/app/data/config/app.env`

这样 Docker 备份、迁移、恢复都会简单很多。

## 9. 数据模型重构建议

V2 不建议继续以 `atlases` 为中心，而建议引入以下核心表。

### 9.1 用户与系统

- `users`
  - 用户账号
- `app_settings`
  - 系统配置
- `prompt_presets`
  - 后台预设提示词模板

### 9.2 猫主体

- `cats`
  - 一只猫一条记录
  - 字段建议：
    - `id`
    - `user_id`
    - `cat_no`
    - `name`
    - `stage`
    - `feed_count`
    - `max_feed_count`
    - `energy`
    - `wisdom`
    - `grit`
    - `creativity`
    - `agility`
    - `cooperation`
    - `overall_power`
    - `appearance_summary`
    - `story_summary`
    - `persona_prompt`
    - `final_image_url`
    - `is_published`
    - `created_at`
    - `updated_at`

### 9.3 喂养记录

- `cat_feed_records`
  - 每喂一次记录一次
  - 字段建议：
    - `id`
    - `cat_id`
    - `feed_index`
    - `source_url`
    - `canonical_url`
    - `video_title`
    - `video_author`
    - `video_summary`
    - `tag_summary`
    - `energy_delta`
    - `wisdom_delta`
    - `grit_delta`
    - `creativity_delta`
    - `agility_delta`
    - `cooperation_delta`
    - `model1_output`
    - `created_at`

### 9.4 对话记录

- `cat_conversations`
  - 会话
- `cat_messages`
  - 消息明细
  - 如果消息中有抖音链接，记录 `triggered_feed_record_id`

### 9.5 生成结果

- `cat_generations`
  - 最终形象生成任务
  - 记录模型 2 / 模型 3 原始输出、图传结果、失败原因

### 9.6 广场

- `plaza_posts`
  - 发布到广场的猫
- `plaza_comments`
  - 广场评论
- `plaza_likes`
  - 广场点赞

## 10. 现有项目到 V2 的映射建议

现有项目不是完全推倒重来，可以复用一部分能力。

### 10.1 可以复用

- 用户注册/登录/管理员后台
- 抖音链接提取与解析逻辑
- 模型配置与图床配置
- 图片上传链路
- `data/vid2cat.db` 的持久化方案
- Docker 单卷挂载思路

### 10.2 需要弱化或迁移

- `atlases`
  - 从核心业务对象降级为历史兼容对象，后续可废弃
- 当前“图鉴详情页”
  - 改造成“我的猫详情页”
- 当前“评论+评分”
  - 改造成“广场评论+点赞”

### 10.3 需要新增

- 猫主体表
- 喂养记录表
- 对话表
- 最终生成表
- 广场发布表

### Phase 2：五次成长闭环

### Phase 4：猫咪广场

- 发布

## 13. 本次设计结论

基于 `项目设计书2.md`，V2 应明确为一个“猫咪养成 + 抖音内容驱动成长 + 广场分享”的产品，而不再是单纯的视频分析图鉴系统。

最关键的三条落地原则：

- 业务中心从 `atlas` 切换到 `cat`
- 所有运行期数据统一收纳到 `data/`
- Docker 只需要挂载 `./data:/app/data` 就能完整保留业务数据

如果按这个方向继续开发，现有 `vid2cat` 的账号体系、抖音解析、模型调用和图床能力都还能继续复用，但数据库模型和页面主流程需要做一次较明显的 V2 重构。
